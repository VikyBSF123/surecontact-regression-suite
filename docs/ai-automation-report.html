💬 How It Was Done

The entire suite was built through natural language conversation with Claude Code. Playwright MCP (Model Context Protocol) allowed Claude to open a real browser, navigate the live QA environment, inspect UI elements, and write accurate selectors — without access to the source code.

  • 1
    Set up Claude Code + Playwright MCP
    Gave Claude a browser to work with. It navigated the live QA site and discovered all pages, flows, and UI patterns on its own.
  • 2
    Described the goal, not the implementation
    Prompt: "Build a complete professional Playwright regression suite for this app." Claude proposed the architecture — POM, fixtures, global setup/teardown, tagging strategy — and executed it.
  • 3
    Iterated in layers
    Each session added a new capability: first the UI tests, then API tests, then mocked error states, then visual/accessibility/performance/security, then tooling and docs.
  • 4
    Reviewed and guided, never coded
    The human role was direction, review, and decision-making. Every line of code was written by Claude. Cursor or VS Code showed the diffs; Claude applied them directly.

📋 How to Replicate This for Your Project

Prerequisites

  • ✓ Running QA/staging web application with login
  • ✓ Node.js 18+ installed (node --version)
  • ✓ Claude Code access (claude.ai/code or VS Code extension)
  • ✓ Playwright MCP configured in Claude settings
  • ✓ Dedicated test account (separate from real users)

Configure Playwright MCP

Add this to your Claude Code MCP settings (claude_desktop_config.json):

// Claude settings → MCP Servers
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}

The Starting Prompt

Once Claude Code is open on an empty folder, use this as your opening message:

"I want to build a complete professional Playwright regression test suite
for my application. The QA URL is [your-url]. Login credentials are
[email] / [password]. Start by exploring the app live using Playwright MCP,
then build the full suite following professional QA engineering standards:
Page Object Model, custom fixtures, API seeding, global setup/teardown,
custom matchers, ESLint, Prettier, Husky, Monocart reporting, and full docs."

Key Patterns to Always Ask For

  • ✓ All test emails use a dedicated domain (e.g. @qatest.io) — global teardown can clean them up
  • ✓ beforeAll API seeding + afterAll cleanup for isolated tests (never depend on live data)
  • ✓ expect.soft() for layout checks — all failures reported together
  • ✓ test.step() on every multi-action test — readable report timeline
  • ✓ Import from fixtures.js not @playwright/test directly
  • ✓ Tag every test.describe — enables targeted runs (@smoke, @critical, etc.)
  • ✓ global-setup.js pre-flight health check — catch broken environments before 300 tests fail
